★ FeaturedSTM32 Cortex-M4

STM32 UART Bootloader

UART bootloader for in-field firmware updates on ARM Cortex-M4 — flash erase, CRC32 verification, safe application handoff via VTOR relocation.

  • CRC32-validated firmware updates over UART
  • Custom linker scripts for bootloader / app memory isolation
  • Safe jump: disable IRQs, stop SysTick, VTOR relocation, MSP reinit
  • Python host tool for firmware streaming and recovery
CSTM32CubeIDEPythonGDBLinker Scripts

How it works end-to-end:

On power-up, the bootloader waits for a host command over UART. Each command packet is CRC32-verified before execution. For a firmware update: erase target flash sector → write binary page-by-page (each page CRC-checked) → verify final image → jump to app.

The jump sequence: disable all IRQs → stop SysTick → relocate vector table via VTOR → reinitialize MSP from app's vector table → branch to reset handler. Bootloader sits at 0x08000000 (start of STM32 flash), application at a higher offset (e.g. 0x08010000).

KEY CHALLENGE: The jump sequence has to be exact — any missed step (VTOR, MSP, IRQ state) causes a hard fault on the other side.

Engineering Takeaways:

  • Precise reset/handoff sequence matters; even minor omissions trigger hard faults.
  • Developing the Python host tool early made field testing far more reliable.

★ FeaturedFreeRTOSSTM32 Cortex-M0

FreeRTOS EDF Scheduler

Extended FreeRTOS to implement Earliest Deadline First scheduling — modified core kernel files to replace the default priority-based scheduler with deadline-ordered execution.

  • Modified tasks.c, tasks.h (kernel files) and FreeRTOSConfig.h
  • Deadline-sorted ready list with dynamic absolute deadline computation
  • Custom xTaskCreatePeriodic() API for periodic task registration
  • Identified first deadline-miss ticks via UART context-switch traces
CFreeRTOSSTM32CubeIDE

What was actually changed inside FreeRTOS:

FreeRTOS uses a fixed-priority ready list. The EDF extension replaces this with a deadline-sorted list: tasks.c was modified so tasks are inserted in order of absolute deadline (release time + period). The scheduler always picks the head — earliest deadline first.

A custom xTaskCreatePeriodic() API creates tasks with a period and relative deadline. Each activation recomputes the absolute deadline. For overloaded task sets (utilization > 1.0), the first deadline-miss tick was captured via UART logging.

KEY INSIGHT: EDF is theoretically optimal for preemptive uniprocessor scheduling — no algorithm can meet deadlines that EDF misses.

Engineering Takeaways:

  • Computing absolute deadlines correctly at task creation is critical to correctness.
  • UART-based context‑switch tracing proved invaluable for observing deadline misses.

★ FeaturedRaspberry Pi 4Kernel Module

Linux Kernel Module — Real-Time GPIO

Replaced a polling-based GPIO loop (consuming ~100% of one CPU core) with an interrupt-driven Linux kernel module — dropping CPU overhead to near-zero with microsecond-level response times.

  • CPU utilization: polling ~100% → interrupt-driven near-zero
  • Kernel interrupt handler via request_irq() on GPIO IRQ line
  • Nanosecond timestamping via hrtimer, latency validated at <10µs
  • Kernel space eliminates user-space scheduling jitter and syscall overhead
CLinux KernelhrtimerRaspberry Pi 4

Why polling was so bad, and how the kernel module fixed it:

The polling approach spun in a tight loop reading the GPIO register — consuming a full CPU core continuously. The kernel module registers an interrupt handler via request_irq() on the GPIO pin's IRQ line. The CPU idles between events; the interrupt fires only on signal change.

High-resolution timers (hrtimer) timestamp interrupt arrival with nanosecond resolution. The delta between physical signal edge and kernel handler entry was consistently in the low microseconds — kernel space removes the scheduling jitter inherent to user-space polling.

KEY RESULT: CPU load ~100% → near-zero between events. Handler latency <10µs, confirmed via logic analyzer.

Engineering Takeaways:

  • Switching from user‑space polling to a kernel interrupt handler freed an entire core.
  • High‑resolution timers plus a logic analyzer make latency measurement painless.

★ FeaturedXinu OS

Process Scheduling in Xinu OS

Extended the Xinu OS kernel with two starvation-free schedulers — an aging-based priority scheduler and a Linux 2.2–style epoch scheduler, both integrated into kernel context switching.

  • Aging scheduler: dynamic priority increments prevent starvation
  • Epoch scheduler: goodness = counter + priority, with quantum carry-over
  • Modified resched.c, ready.c, create.c, proc.h at kernel level
  • Runtime policy selection via setschedclass() / getschedclass()
CXinu OSKernel Scheduling

Two schedulers, two problems:

Aging: Under strict priority, low-priority processes starve. The fix: at every reschedule event, increment each waiting process's effective priority. Once it reaches the current highest, it runs, then resets. Modified resched.c and proc.h to hold the age counter per PCB.

Epoch (Linux 2.2): Each epoch assigns quantum = base + priority. Unused quantum carries over: new = base + priority + leftover/2. Goodness = counter + priority determines who runs. New epoch starts when all counters hit zero. Modified resched.c, ready.c, create.c, proc.h.

★ FeaturedXinu OSSynchronization

Readers-Writer Locks with Priority Inheritance

Kernel-level readers-writer lock system in Xinu OS with a full priority inheritance protocol — solving priority inversion with transitive propagation across lock chains.

  • R/W lock semantics: concurrent readers, exclusive writers, priority-ordered wait queues
  • Priority inheritance: lock holder's priority raised to match highest-priority waiter
  • Transitive inheritance: propagates across chains (A waits on B waits on C → all elevated)
  • Fixed Xinu's sdelete bug — waiting processes now return DELETED, not OK
CXinu OSKernel SynchronizationReal-Time Systems

Why this matters in real-time systems:

Priority inversion — where a high-priority task is blocked by a low-priority one holding a lock — can cause timing violations in real-time systems. The fix: when a high-priority process P1 waits on a lock held by lower-priority P2, P2's effective priority is raised to match P1's. If P2 is itself blocked waiting on another lock held by P3, P3's priority is raised too (transitive propagation).

Implementation: Built linit.c, lcreate.c, ldelete.c, lock.c, and releaseall.c as a new subsystem alongside (not modifying) the existing Xinu semaphore system. The lock table tracks current holders, wait queue priorities, and inherited priorities per process. releaseall() atomically releases multiple locks and recalculates inherited priorities.

Locking policy: Readers are admitted if no writer holds the lock and no equal-or-higher-priority writer is waiting. Writers are always given preference at equal priority. Ties broken by wait time.

KEY INSIGHT: Priority inversion is a real failure mode in embedded RTOS systems — the Mars Pathfinder mission famously suffered from it in 1997.

CompilersLLVM

LLVM IR Optimizer

Custom LLVM IR optimizer implementing 4 optimization passes on SSA-form IR — the same class of transformations that GCC and Clang apply to embedded firmware.

  • Global CSE via dominator tree analysis
  • Dead code elimination — removes instructions with no live uses
  • Store-to-load forwarding & redundant store elimination
  • Per-pass statistics and IR verification
C++LLVMSSADominator Trees

What each pass does:

Global CSE: Builds a dominator tree. If an expression was computed in a dominating block and operands are unchanged, the recomputation is replaced with the existing value.
DCE: Removes instructions whose results have no live uses reachable from the definition.
Store-to-load forwarding: If a store to X is followed by a load from X with no intervening store, the load is replaced directly with the stored value.
Redundant store elimination: Two stores to the same address with no intervening load — the first is dead and removed.

Engineering Takeaways:

  • IR verification after each pass caught subtle bugs early.
  • Collecting per‑pass statistics helped tune optimization order and aggressiveness.
